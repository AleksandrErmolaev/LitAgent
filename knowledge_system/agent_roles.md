# Роли агентов в мультиагентной системе анализа литературы

## Общая архитектура

Система состоит из семи специализированных агентов, взаимодействующих через REST API либо напрямую через Python-вызовы (в рамках единого контейнера при тесной интеграции). Каждый агент реализован как stateless-сервис, за исключением компонентов кэширования (Booker). Оркестрация выполняется центральным Orchestrator Agent, а оценка качества результатов — Evaluator Agent с привлечением статической базы знаний и RAG.

Агенты и их роли:
- **Booker** — поиск и загрузка полного текста книги.
- **CharacterAnalyzer** — выделение персонажей, определение их ролей, архетипов, черт и связей.
- **Summarizer** — многоуровневое реферирование текста и выделение ключевых цитат.
- **QuestionGenerator** — генерация вопросов разной сложности для проверки понимания текста.
- **Reporter** — сборка итогового отчёта в форматах JSON и Markdown.
- **Orchestrator** — координация выполнения пайплайна, обработка ошибок, валидация форматов.
- **Evaluator** — оценка качества артефактов, сгенерированных другими агентами, по формальным критериям.

## 1. Booker Agent

### Назначение
Поиск и загрузка полного текста литературного произведения по названию и автору из открытых источников: Project Gutenberg, Wikisource, Google Books, LitRes (при наличии API-ключа). Агент использует внутренний кэш на базе Redis для ускорения повторных запросов.

### Входные данные
```json
{
  "title": "string",
  "author": "string",
  "language": "ru" | "en",
  "source_hint": "string (optional)"
}
```

### Выходные данные (успех)
```json
{
  "status": "success",
  "title": "string",
  "author": "string",
  "language": "string",
  "text": "string",
  "word_count": "int",
  "chapters": ["string"],
  "source": "gutenberg" | "wikisource" | "google_books" | "litres",
  "cached": "boolean",
  "processing_time_ms": "int"
}
```

### Выходные данные (ошибка)
```json
{
  "status": "error",
  "error_code": "BOOK_NOT_FOUND" | "MISSING_TITLE" | "SOURCE_UNAVAILABLE",
  "message": "string",
  "attempted_sources": ["string"]
}
```

### Эндпоинты
- `POST /fetch` — возвращает метаданные книги без полного текста.
- `POST /fetch_full` — возвращает полный текст вместе с метаданными (может быть объёмным).
- `GET /health` — проверка состояния сервиса.
- `DELETE /cache/{title}/{author}` — сброс кэша для конкретной книги.

## 2. CharacterAnalyzer Agent

### Назначение
Анализ полного текста с целью выявления персонажей, их характеристик и взаимосвязей. Использует spaCy для NER (извлечение имён персонажей) и LLM для генерации описаний, архетипов, черт и репрезентативных цитат. Строит граф отношений на основе совместной встречаемости персонажей в абзацах.

### Входные данные
```json
{
  "full_text": "string"
}
```

### Выходные данные
```json
{
  "characters": [
    {
      "name": "string",
      "role": "string (e.g., protagonist, antagonist, secondary, minor)",
      "archetype": "string (hero, mentor, trickster, lover, innocent, ruler, creator, caregiver, everyman, anti-hero)",
      "traits": ["string"],
      "description": "string",
      "mentions_count": "int",
      "quote": "string"
    }
  ],
  "relationships": [
    {
      "from": "string",
      "to": "string",
      "type": "string (e.g., associated_with, loves, conflicts_with)",
      "strength": "float (0.0 to 1.0)"
    }
  ]
}
```

### Примечания
- Минимальное количество упоминаний для включения персонажа в результат по умолчанию — 5.
- Для работы агента требуется предустановленная spaCy-модель (например, `en_core_web_lg`). В контейнере модель загружается при сборке.
- В случае отсутствия LLM-клиента агент возвращает только базовую информацию без сгенерированных полей.

### Эндпоинты
- `POST /analyze` — принимает полный текст, возвращает результат анализа.
- `GET /health` — статус сервиса.

## 3. Summarizer Agent

### Назначение
Создание трёхуровневой суммаризации литературного произведения:
- `super_short` — 1–3 предложения, передающие основную идею и ключевой конфликт.
- `short` — абзац объёмом до 150 слов, охватывающий завязку, развитие и развязку.
- `detailed` — подробное изложение сюжета объёмом 300–500 слов с сохранением последовательности событий.

Дополнительно агент выделяет ключевые цитаты (до 5) и пишет краткие аннотации для каждой главы.

### Входные данные
```json
{
  "full_text": "string",
  "chapters": ["string"]
}
```

### Выходные данные
```json
{
  "super_short": "string",
  "short": "string",
  "detailed": "string",
  "key_quotes": ["string"],
  "chapter_summaries": ["string"]
}
```

### Примечания
- При отсутствии разбиения на главы агент выполняет суммаризацию всего текста целиком, а `chapter_summaries` остаётся пустым.
- Для генерации используется LLM с разными параметрами `max_tokens` в зависимости от уровня детализации.

### Эндпоинты
- `POST /summarize` — основной метод суммаризации.
- `GET /health` — проверка работоспособности.

## 4. QuestionGenerator Agent

### Назначение
Генерация набора вопросов разной сложности для проверки понимания прочитанного текста. Каждый вопрос сопровождается четырьмя вариантами ответа (один правильный, три правдоподобных дистрактора), цитатой-подтверждением из текста и указанием главы, к которой относится вопрос.

### Входные данные
```json
{
  "full_text": "string",
  "chapters": ["string"],
  "characters": [
    {
      "name": "string",
      "role": "string",
      ...
    }
  ]
}
```

### Выходные данные
```json
{
  "questions": [
    {
      "text": "string",
      "difficulty": "easy" | "medium" | "hard",
      "options": ["string", "string", "string", "string"],
      "correct_answer": "string",
      "quote_evidence": "string",
      "chapter": "int"
    }
  ]
}
```

### Классификация сложности
- **easy** — фактический вопрос, ответ на который явно присутствует в тексте (например, «Как звали главного героя?»).
- **medium** — требует понимания причинно-следственных связей или интерпретации поступков персонажа.
- **hard** — предполагает анализ идейного содержания, темы, символики.

### Эндпоинты
- `POST /generate` — возвращает сгенерированные вопросы.
- `GET /health` — статус.

## 5. Reporter Agent

### Назначение
Сборка итогового отчёта, объединяющего результаты работы всех предыдущих агентов. Отчёт предоставляется в двух форматах: структурированный JSON (для машинной обработки) и человеко-читаемый Markdown (для просмотра и публикации).

### Входные данные
Агент ожидает на входе агрегированный словарь, соответствующий модели `ReportInput` (Pydantic):
```json
{
  "summary": {
    "super_short": "string",
    "short": "string",
    "detailed": "string",
    "key_quotes": ["string"],
    "chapter_summaries": {"Chapter 1": "string", ...}
  },
  "characters": {
    "characters": [ ... ],
    "relationships": [ ... ]
  },
  "questions": {
    "questions": [ ... ]
  },
  "metadata": {
    "title": "string",
    "author": "string",
    "language": "string",
    "word_count": "int",
    "year": "string (optional)",
    "source_url": "string (optional)"
  },
  "model_used": "string",
  "processing_time_ms": "int"
}
```

### Выходные данные
```json
{
  "json_report": { ... },      
  "markdown_report": "string"  
}
```

### Эндпоинты
- `POST /report` — генерация отчёта (в реальной системе агент может вызываться напрямую из Orchestrator, без отдельного HTTP-эндпоинта).
- `GET /health` — статус.

## 6. Orchestrator Agent

### Назначение
Центральный координатор мультиагентной системы. Отвечает за:
- Приём исходного запроса от пользователя (название, автор книги).
- Последовательный вызов агентов в правильном порядке.
- Обработку ошибок на каждом этапе (например, если книга не найдена — информирование пользователя и прекращение пайплайна).
- Валидацию выходных форматов агентов согласно контрактам, описанным в `agent_roles.md`.
- Принятие решений о повторных попытках (до 2–3 раз) при временных сбоях источников.
- Передачу итогового агрегированного результата в Reporter и возврат конечного ответа пользователю.

### Входные данные (пользовательский запрос)
```json
{
  "title": "string",
  "author": "string",
  "language": "ru" | "en"
}
```

### Выходные данные (конечный ответ)
Итоговый отчёт в формате, сгенерированном Reporter Agent:
```json
{
  "json_report": { ... },
  "markdown_report": "string"
}
```
При возникновении невосстановимой ошибки возвращается структура с полем `error`.

### Внутренняя логика
1. Вызвать **Booker** для получения полного текста и разбиения на главы.
2. Параллельно (если возможно) или последовательно вызвать:
   - **CharacterAnalyzer** для извлечения персонажей.
   - **Summarizer** для получения суммаризации.
3. Передать результаты CharacterAnalyzer и Summarizer в **QuestionGenerator**.
4. Собрать все результаты и метаданные, вызвать **Reporter**.
5. (Опционально) Вызвать **Evaluator** для проверки качества артефактов и при необходимости инициировать доработку (перегенерацию отдельных частей).

### Примечания
- Orchestrator не имеет собственного HTTP-интерфейса, если система запускается как единое приложение; однако может быть оформлен как отдельный сервис с эндпоинтом `/analyze`.
- Использует файл `agent_roles.md` (через RAG) для проверки соответствия форматов.

## 7. Evaluator Agent

### Назначение
Оценка качества артефактов, сгенерированных другими агентами (суммаризации, вопросов, анализа персонажей), по заранее определённым критериям. Использует:
- Статическую базу знаний (`evaluation_criteria.md`).
- Few-shot примеры (`few_shot_examples_summarizer.md`, `few_shot_examples_qa.md`).
- При необходимости — извлечение релевантных фрагментов исходного текста для проверки фактов.

Evaluator может вызываться как в процессе пайплайна (для принятия решения о перегенерации), так и постфактум для независимого аудита качества.

### Входные данные
```json
{
  "artifact_type": "summary" | "questions" | "characters",
  "artifact": { ... },          
  "source_text": "string (optional)",
  "context": {}                   
}
```

### Выходные данные
```json
{
  "score": "float (0.0–10.0)",
  "passed": "boolean",          
  "feedback": "string",         
  "detailed_metrics": {          
    "completeness": "float",
    "accuracy": "float",
    "style": "float",
    ...
  }
}
```

### Критерии оценки (кратко)
- **Суммаризация**: полнота, сжатость, точность, стиль, качество цитат.
- **Вопросы**: релевантность, корректность ответа, сложность, дистракторы, цитата.
- **Персонажи**: полнота выделения, роль, архетип, черты, описание, связи.

Подробные рубрики и примеры оценок приведены в файле `evaluation_criteria.md`.

### Эндпоинты
- `POST /evaluate` — оценка артефакта.
- `GET /health` — статус.
